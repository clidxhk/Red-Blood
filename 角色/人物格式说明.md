# 人物谱牒写法

[[东阳]] [[世界背景]] [[赤龙]]

## 概述

人物条目以 Markdown 为正篇，以 JSON 为底谱。

前者负责让读者真正“见到”一个人，能读出气息、身世、伤痕、关系与命运；后者负责把那些会反复调用的要素沉成稳定结构，方便后续扩写、检索、复用与批量校对。两者不争位置，而是分前台与后台。

## 核心设定

一篇成熟的人物条目，最好拆成两层:

- `md` 层承载叙事与氛围。重点写观感、因果、气味、动作、人物与世界的勾连。
- `json` 层承载结构与索引。重点写字段边界、枚举关系、阶段节点、可复用标签。

这样做的好处，是既不会把人物写成冷冰冰的表格，也不会让关键细节淹没在大段抒写里难以抽取。

## Markdown 层

建议固定为以下六段，不必每次逐字照搬，但职责尽量稳定:

### 一、概述

用一到三段立住人物的整体气象，不交代履历清单，而先让读者知道这人站出来时，天地会变成什么样子。

然后描述人格类型，这样的好处是，给每个人物一个性格，使得人物更立体。

### 二、核心设定

直接点明此人之于世界观的功能。例如他代表哪一股时代力量、修行逻辑补上了什么空缺、与旧秩序的根本冲突在哪里。

### 三、身世与行历

按因果写，不按流水账写。重要的不是“几年做了什么”，而是“哪件事把他推成了现在这个样子”。

### 四、修行与能力

不只列招式名称。每个核心能力最好同时带上:

- 它解决什么问题
- 它呈现什么视觉或感官印象
- 它潜藏什么代价或边界

### 五、关系与位置

人物不是孤岛。要说明他和同道、对手、宗门、法统、地理、时代之间的连接线。

### 六、传闻与遗音

用一两则小事或一两句余音收尾，比单独堆“名言”更容易留下人物余温。

## 设计原则

### 一、让 JSON 记录“稳定事实”，让 MD 负责“表现事实”

例如“他衣上有潮木和药灰气息”这种可复用细节，适合进 JSON 的 `scent`；至于那气息在雨夜里如何让人联想到旧炉堂，就留给 MD 去展开。

### 二、让字段直接服务扩写

`turning_points`、`scene_seeds`、`signature_methods` 这类字段，都是为了后续写场景、写支线、写回忆时可直接调用，而不是为了把人物资料塞满。

### 三、避免现实映射裸露在表层

即便人物确实借现代历史为骨，也尽量把 JSON 字段写成世界内语言。保留“时代角色”“权力逻辑”“阵营关系”，不要保留“现实原型是谁”的明示标签。

### 四、把“代价”写进结构

人物一旦只有能力和光环，神秘感会很快变廉价。JSON 很适合专门收纳限制、代价、禁忌、失控条件，这些信息在正文里不一定每次都展开，但在设定层必须存在。

## 适用方式

后续新建人物时，可以先写 `json` 骨架，明确边界，再写 `md` 正文；也可以先把 `md` 写成熟，再回填 `json` 结构。若人物定位复杂，优先先定 `json`，避免正文越写越散；若人物气质难抓，优先先写 `md`，再把稳定设定沉淀成结构。
